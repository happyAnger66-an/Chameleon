# 机器人端侧模型推理部署：传统特点与 Agent 化发展趋势

本文基于 TensorRT-Edge-LLM 与端侧 Agent 发展趋势，系统性总结机器人端侧模型推理部署的传统特点、Agent 化带来的新特点，以及由此产生的工程难点。

## 1. 背景

机器人端侧模型部署正在从“固定任务推理”演进为“长期运行的本地 Agent 系统”。

传统形态更接近嵌入式推理 pipeline：

```text
传感器输入 -> 预处理 -> 单个/少数模型推理 -> 后处理 -> 控制或输出
```

Agent 化后的形态更复杂：

```text
观察环境 -> 理解目标 -> 规划步骤 -> 调用工具/技能 -> 执行动作 -> 观察反馈 -> 修正计划
```

因此，机器人端侧部署的核心问题不再只是“模型能否在 Jetson/DRIVE 上跑起来”，而是“如何在有限算力、显存、功耗和实时约束下，让多个模型长期、安全、稳定地协同运行”。

## 2. 传统机器人端侧推理部署特点

传统机器人端侧推理通常围绕固定任务展开，例如目标检测、语音识别、导航、抓取识别、状态分类等。

| 维度 | 传统特点 |
| --- | --- |
| 任务形态 | 固定任务为主，输入输出明确 |
| 模型类型 | CNN、检测模型、ASR、小型 VLM/LLM、策略网络 |
| 推理模式 | 单次请求或短链路 pipeline |
| 上下文 | 短上下文，通常只关心当前帧、当前语音、当前指令 |
| 部署目标 | 低延迟、低功耗、稳定运行 |
| Runtime | C++、TensorRT、ONNX Runtime 或自研推理框架 |
| 数据流 | 静态、明确、工程可控 |
| 安全边界 | 主要关注传感器异常、控制边界、急停机制 |

以 TensorRT-Edge-LLM 类框架为例，传统部署路径通常是：

```text
HuggingFace Checkpoint
  -> 量化
  -> ONNX 导出
  -> Edge 设备上构建 TensorRT engine
  -> C++ runtime 推理
  -> 机器人应用调用
```

这种方式强调的是确定性和可控性：模型、输入、输出、延迟、显存占用和功耗都可以比较清楚地 profile 和验证。

## 3. 传统部署的工程优势

传统机器人端侧推理部署有几个明显优势：

- 链路确定：输入输出格式固定，调试路径较短。
- 实时性可控：每个模型的 latency、显存、功耗可以独立评估。
- 安全边界清晰：模型多用于感知、分类或局部策略，较少直接做开放式决策。
- 资源预算明确：可以提前确定每个模型占用的 GPU、CPU、内存和带宽。
- 部署形态简单：通常是一个或几个 TensorRT engine 加 C++ 控制逻辑。
- 测试可重复：固定数据集、固定场景、固定指标，便于回归测试。

因此，传统端侧模型部署更像嵌入式系统工程，关键词是：

```text
确定任务、固定 pipeline、TensorRT engine、低延迟、低功耗、稳定性
```

## 4. Agent 化带来的新特点

机器人 Agent 不再只是识别一个物体或执行一条语音命令，而是要处理开放式、多步骤、可恢复的任务。

典型任务可能是：

```text
“帮我把桌上的红色杯子拿到厨房，如果路上有人挡住，就绕过去。”
```

这类任务涉及语言理解、视觉感知、任务规划、导航、抓取、异常恢复和人机交互。部署系统也因此出现一系列新特点。

### 4.1 从单模型变成多模型协同

传统部署常见形态是单模型或少数模型串联：

```text
图像/语音/指令 -> 模型推理 -> 输出结果
```

Agent 化后，机器人端侧往往需要多个模型和模块协同：

```text
ASR -> LLM/VLM -> Planner -> Skill Selector -> Action Model -> TTS
              ^        |
              |        v
        Memory / Retrieval / Tools / Sensors
```

常见组件包括：

- 语音模型：ASR、TTS、语音活动检测。
- 视觉模型：检测、分割、VLM、视频理解。
- 语言模型：指令理解、对话、任务规划。
- Embedding 模型：本地记忆检索。
- 动作模型：抓取、导航、机械臂控制、VLA。
- 安全模型：动作约束、异常检测、碰撞风险判断。

部署难点从“优化一个模型”变成“调度一组模型”。

### 4.2 从短上下文变成长生命周期状态

传统上下文通常是：

```text
当前图像帧 + 当前语音 + 当前指令
```

Agent 上下文则包括：

```text
当前任务
+ 历史对话
+ 用户偏好
+ 物体位置
+ 地图状态
+ 工具调用结果
+ 失败尝试记录
+ 机器人自身状态
+ 环境变化
```

机器人端侧 Agent 需要更长上下文，但端侧显存有限，不能简单依赖超大 token window。更现实的方式是：

```text
KV cache + 摘要 + 本地向量库 + 状态数据库 + 任务图
```

也就是说，长上下文会从“单纯扩大 prompt”变成“分层记忆系统 + 选择性上下文装载”。

### 4.3 从被动响应变成持续运行

传统推理通常是事件触发：

```text
收到输入 -> 推理 -> 输出 -> 结束
```

Agent 是循环式运行：

```text
观察 -> 思考 -> 行动 -> 再观察 -> 修正 -> 继续行动
```

这要求端侧部署系统支持：

- 模型长期常驻内存。
- 低功耗 idle 模式。
- 快速唤醒。
- 任务中断与恢复。
- 状态持续维护。
- 显存和 KV cache 动态回收。

部署系统需要从“单次 inference runtime”升级为“长期运行的 Agent runtime”。

### 4.4 从纯推理变成工具和技能调用

机器人 Agent 会调用真实世界工具和技能，例如：

- 移动底盘。
- 机械臂。
- 摄像头云台。
- 抓手。
- 地图导航。
- 语音播报。
- 本地数据库。
- Web/API 服务。
- 家居或工业设备接口。

模型输出不再只是文本，而可能是结构化动作：

```json
{
  "skill": "pick_object",
  "target": "red cup",
  "location": "table_left"
}
```

因此部署系统需要把模型输出接到可验证、可约束、可回滚的执行层。

### 4.5 从本地推理变成本地优先的端云协同

机器人端侧 Agent 倾向本地处理：

- 家庭、工厂、车内等隐私数据。
- 避障、抓取、语音交互等实时任务。
- 无网络或弱网络环境下的离线任务。

但复杂规划、知识密集推理或超大模型能力仍可能依赖云端。因此未来常见形态会是：

```text
端侧小模型常驻
+ 本地记忆和工具
+ 云端大模型可选增强
+ 策略路由决定是否上云
```

这要求系统具备任务分级、隐私过滤、云端回退和离线降级能力。

## 5. Agent 化带来的新难点

### 5.1 资源调度难度显著上升

机器人端侧资源有限：

- GPU 显存有限。
- CPU 同时运行感知、控制、中间件和业务逻辑。
- 内存带宽有限。
- 功耗和散热受限。
- 电池供电场景对能耗敏感。

Agent 化后，多模型会同时争抢资源：

```text
VLM 占 GPU
ASR/TTS 占 GPU/CPU
LLM decode 占显存和内存带宽
导航/SLAM 占 CPU/GPU
控制环要求实时
```

系统需要动态决策：

- 哪些模型常驻？
- 哪些模型按需加载？
- KV cache 保留多久？
- 视觉帧采样频率如何调整？
- 电量或温度异常时如何降级？
- 控制任务和语言任务谁优先？

### 5.2 长上下文与有限显存矛盾

Agent 需要长期记忆，但端侧无法像数据中心一样通过堆显存解决问题。

主要挑战包括：

- KV cache 占用持续增长。
- 多轮任务导致上下文膨胀。
- 多模态 token 成本高。
- 视频和图像上下文难以长期保留。
- 摘要压缩可能丢失关键信息。

未来需要更多工程机制：

- Prefix cache / system prompt cache。
- FP8/INT8 KV cache。
- 分层 memory。
- 局部检索。
- 任务摘要。
- 环境状态结构化。
- 过期信息淘汰。

### 5.3 实时控制与大模型推理冲突

机器人存在硬实时或准实时链路：

- 避障。
- 平衡控制。
- 机械臂轨迹控制。
- 碰撞检测。
- 人机交互响应。

LLM/VLM 推理延迟通常较高且波动较大，不能直接阻塞控制环。合理架构应分层：

```text
低层控制：实时、确定性，使用传统算法或小模型
中层技能：可验证动作 primitive
高层 Agent：慢速规划、语言理解、任务分解
```

高层 Agent 应下发目标和技能调用，而不是直接控制每个电机动作。

### 5.4 安全问题从“模型错了”变成“机器人做错了”

传统模型出错，可能只是识别错一个物体。Agent 出错，可能触发真实世界风险：

- 抓错物体。
- 撞到人或设备。
- 进入危险区域。
- 执行未授权动作。
- 泄露摄像头或语音隐私。
- 被 prompt injection 诱导执行危险命令。

部署系统需要具备安全执行机制：

- 动作权限系统。
- 高风险动作二次确认。
- 物理安全边界。
- 工具调用 sandbox。
- 输出 schema 校验。
- Skill-level guardrail。
- 本地数据访问控制。
- 日志与审计。

### 5.5 多模态数据管理更复杂

机器人面对的是连续物理世界，而不是单一文本输入。数据来源包括：

- 视频流。
- 深度图。
- IMU。
- 关节状态。
- 力反馈。
- 地图。
- 语音。
- 人的位置和手势。

难点在于如何选择、压缩和结构化这些信息：

- 不能把每一帧都送入 VLM。
- 需要关键帧选择。
- 需要事件触发。
- 需要连续状态结构化。
- 需要物体级世界模型。
- 需要同步不同传感器时间戳。

未来机器人端侧部署会更依赖 world model、scene graph 和 object memory，而不是单纯依赖 token prompt。

### 5.6 可测试性变差

传统模型可以用固定测试集评估：

```text
准确率、延迟、吞吐、显存
```

Agent 任务则是开放式、多步骤、环境相关的：

```text
理解目标 -> 规划 -> 导航 -> 抓取 -> 失败恢复 -> 交互确认
```

测试难点包括：

- 任务路径不唯一。
- 错误可能发生在任意步骤。
- 模型输出存在随机性。
- 真实环境难以复现。
- 多模型组合导致责任归因困难。

需要新的测试体系：

- 仿真环境。
- 场景回放。
- 行为轨迹评估。
- 工具调用日志检查。
- 长任务成功率。
- 安全违规率。
- 人机交互延迟。

## 6. 未来机器人端侧推理部署架构趋势

未来机器人端侧部署大概率会形成分层架构：

```text
+------------------------------------------------+
| 高层 Agent                                      |
| LLM/VLM、任务规划、对话、长期记忆                |
+------------------------------------------------+
| 中层 Skill Runtime                              |
| 导航、抓取、搜索、检查、交互、恢复策略            |
+------------------------------------------------+
| 感知与世界模型                                  |
| VLM、检测、分割、SLAM、场景图、物体记忆          |
+------------------------------------------------+
| 低层实时控制                                    |
| 轨迹规划、避障、关节控制、安全停止               |
+------------------------------------------------+
| Edge Inference Runtime                          |
| TensorRT Engine、量化、KV cache、调度            |
+------------------------------------------------+
```

其中端侧推理框架不再只是执行模型，而要支持：

- 多模型生命周期管理。
- 显存和 KV cache 管理。
- 动态量化和按需加载。
- 工具调用接口。
- 本地 memory store。
- 实时任务优先级。
- 云端协同路由。
- 安全策略执行。

## 7. 传统特点与新特点对比

| 维度 | 传统机器人端侧部署 | Agent 化后的新部署 |
| --- | --- | --- |
| 任务 | 固定、单步 | 开放式、多步、可恢复 |
| 输入 | 当前帧、当前语音、当前命令 | 长历史、多模态、环境状态 |
| 模型 | 单模型或少数 pipeline | 多模型、多 runtime 协同 |
| 上下文 | 短上下文 | 长上下文 + 本地记忆 |
| 推理 | 请求-响应 | 持续循环 |
| 输出 | 分类、检测、文本、简单动作 | 工具调用、技能调用、任务计划 |
| 调度 | 静态配置 | 动态资源调度 |
| 安全 | 传感器和控制安全 | 权限、安全策略、行为审计 |
| 测试 | 数据集评估 | 场景级、任务级评估 |
| 优化目标 | 延迟、显存、功耗 | 延迟 + 记忆 + 安全 + 任务成功率 |

## 8. 对 TensorRT-Edge-LLM 类框架的启示

结合 TensorRT-Edge-LLM 的部署特点，机器人 Agent 趋势会推动端侧推理框架增加以下能力：

- 更强 KV cache 管理：长任务下的 prefix/system prompt cache、KV 量化、KV 淘汰。
- 多模型 runtime 编排：LLM、VLM、ASR、TTS、VLA/action model 统一调度。
- 更丰富的 C++ API：不只是 chat completion，而是面向机器人 skill/action 的结构化输出。
- 流式与中断能力：机器人任务需要边生成边执行，也需要随时取消。
- 低功耗常驻模式：Agent 常驻后台，不能一直满功耗 decode。
- 端云协同接口：本地优先，复杂任务可上云。
- 安全执行层：模型输出必须经过 schema 校验、权限检查和 skill guardrail。
- 多模态状态缓存：图像、场景、物体记忆会比单纯文本 cache 更重要。

## 9. 总结

传统机器人端侧模型部署的关键词是：

```text
确定任务、固定 pipeline、TensorRT engine、低延迟、低功耗、稳定性
```

Agent 发展后的新关键词是：

```text
长上下文、本地记忆、多模型协同、持续运行、工具调用、安全执行、端云协同
```

最大难点不再是“一个模型能不能在 Jetson 上跑”，而是：

```text
在有限算力、显存和功耗下，
让多个模型长期协同运行，
同时保持实时性、安全性、可解释性和任务成功率。
```

因此，机器人端侧部署正在从模型部署问题演进为端侧 Agent 系统工程问题。
